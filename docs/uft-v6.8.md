## 0. 何が変わるのか

v6.2 までで v3.1 の主張は「三重の独立実装 + SHA-256 証明書」まで固めた。しかし
それでも信頼の基盤は「Rust プログラムが正しく書けていること」だった。本バージョンは
探索そのものを **Lean 4 の定理**にする。信頼基盤は Lean の型検査器 (と、下記の通り
一部はコンパイラ) まで縮む。

## 1. 定理

`proofs/Anomaly.lean` は v3.1 の領域 (SU(3) 表現 ≤ 3/3̄・SU(2) ≤ 2・|6Y| ≤ 9・
多重項 ≤ 5・成分 ≤ 15、条件 = 5 アノマリー + Witten + カイラル + 全因子帯電) を
整数データとして形式化し、次を証明する:

| 定理 | 内容 | 検証手段 (信頼基盤) |
|---|---|---|
| `sm_is_solution` | SM 1 世代は全条件を満たす | `decide` (**Lean 核のみ**) |
| `sm_breaks_if_shifted` | e^c の超電荷を 1 目盛ずらすと破れる | `decide` (核のみ) |
| `sm_minimal_unique` | **領域の全列挙で、解はちょうど 4 個 = SM の同値軌道 (恒等/共役/U(1)反転/両方) に一致** | `native_decide` (核 + Lean コンパイラ) |

「解 4 個」は生の多重集合としての数であり、同値で割れば 1 個 — v3.1/v6.2 の
「SM が最小かつ一意」と同じ内容の、より強い形 (軌道の明示) である。

## 2. 設計上の正直な注記

1. **native_decide の信頼基盤**: 全列挙 (~6×10⁷ 節点) は Lean 核の項簡約では遅すぎる
   ため、コンパイル実行で真偽を確定する `native_decide` を使う。これは Lean コンパイラを
   信頼基盤に含める。核のみで検証されるのは定理 1・2。この区別を明記する。
2. **毒値の設計**: 列挙は燃料つき再帰で書かれ、異常は走査結果 (自然数) を毒する —
   SM 軌道に入らない解は +10⁶、燃料切れは +10⁹。したがって定理 `scanResult = 4` の
   成立自体が「(a) 解はちょうど 4 個、(b) 全て SM 軌道、(c) 列挙は完全 (打ち切りなし)」
   の 3 つを同時に含意する (黙った打ち切り・黙った例外で定理が偽になる設計)。
   カイラル条件の単調性 (共役対は要素を足しても消えない) を枝刈りに使い、
   増分和は `checkAll` と同一の式 — `checkAll` 自体は核検証の定理 1・2 が SM 点で照合する。
3. **軌道の有限性**: |6Y| ≤ 9 では SM 電荷 (gcd=1, 最大 |6Y|=6) の非自明な整数倍は
   範囲外なので、同値軌道は 4 通りで尽きる (ソース内に明記)。
4. **形式化と Rust の対応**: 表現テーブル・条件式は v6.2 の Rust 実装と同一の整数
   データであり、Lean は第 4 の独立実装でもある (DFS/MITM/直接 DFS/Lean)。

## 3. 再現手順

```bash
# Lean 4 (elan) が必要: https://leanprover.github.io
~/.elan/bin/lean proofs/Anomaly.lean   # 終了コード 0 = 全定理検証済み
```

`proofs/lean-toolchain` がバージョン (v4.31.0) を固定する。CI にも検証ジョブを追加した。
